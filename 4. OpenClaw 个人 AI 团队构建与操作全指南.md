# OpenClaw 个人 AI 团队构建与操作全指南

## 1. 【工具简介】：OpenClaw 的核心价值与应用场景

在人工智能驱动的效能革命中，传统的“单兵作战”模式正迅速向“多智能体（Multi-Agent）团队协作”转型。这种转变的战略意义在于，通过将复杂的业务逻辑拆解并分配给专业化的 AI 角色，个人能够从琐碎的执行层面抽离，转向更高维度的决策与调度。OpenClaw 正是这一转型中的核心底座，它不仅是一个管理系统，更是个人数字资产的自动化工厂。

### 核心功能总结

OpenClaw 能够将单一的 AI 交互转化为多智能体协同工作流。其核心能力在于管理多个具有独立人格、任务边界和工具权限的 Agent。它支持定时任务触发、24/7 不间断运行以及主从代理间的动态调度，通过去中心化的构建方式，确保任务流转的自动化与闭环。

### 应用场景分析：六大标杆角色

基于 OpenClaw 构建的智能体团队可覆盖以下核心业务领域：

- **情报员 (Gock)：** 每日定时搜寻行业进展与用户痛点。其核心在于“交互式深挖”逻辑：若用户对汇报内容进行反馈，将触发针对特定话题的递归式搜索。
- **私人助理 (Otter)：** 深度集成日程管理、语音备忘及环境信息同步，旨在完全替代 Notion 等传统笔记软件。
- **视频生产流水线 (May)：** 具备最高权限，通过监控热点自动生成翻译与研究报告。其特色在于“人机协作触发制”：用户点击 **[Thumbs Up]** 即可推进研究与脚本生成，点击 **[Eye]** 则将内容存入收藏夹。
- **开发者 (Pandas)：** 承担高价值的编码、自动化测试及边缘情况排查，是团队中主要的 24/7 生产力单元。
- **内容转译 (Monkey)：** 专门负责媒体资产的二次利用，将视频链接一键转化为带截图的深度图文。
- **安全审计 (Tiger)：** 负责系统扫描与版本更新评估，确保新功能加入不会导致性能退化。

### 问题解决能力

该工具通过多代理协作模式，有效解决了个人精力有限、信息过载和任务执行中断的问题。其定时触发与不间断工作的特性，让个人拥有了一支“不睡觉、各司其职”的数字化外挂团队。

在理解了工具的宏观价值后，确保系统稳定运行的第一步是完成精密的前置环境准备。

## 2. 【准备工作/前置条件】：构建团队的基石

构建 AI 团队并非简单的软件安装，而是一次生产环境的配置。良好的准备工作是确保智能体能够稳定接收指令并调用工具的关键。

### 账号与渠道准备

- **Telegram 账号：** Telegram 是 OpenClaw 的主要交互界面与控制台。所有的指令下达、状态反馈及智能体间的通信均通过此渠道实现。

### 凭证获取

- **TOKEN 申请：** 每一个新智能体都需要一个合法的身份凭证。用户必须通过 Telegram 官方的 **[BotFather]** 应用，利用 **[create a new bot]** 指令获取唯一的 **TOKEN**。这是连接 OpenClaw 底层逻辑与前端交互的唯一纽带。

### 环境认知与成本战略

- **运行环境权衡：** 24/7 运行的高价值任务（如开发、销售线索挖掘）需考量成本。使用云端大模型会产生持续的 Token 消耗；若选择本地运行，则需配置如 Mac Studio 等高性能硬件。这种权衡决定了团队的规模与工作负载分配。

基础设施配置完成后，即可进入最核心的实操搭建阶段。

## 3. 【详细操作步骤】：从 0 到 1 搭建你的 AI 团队

OpenClaw 的核心优势在于“通过对话生成工具”，这种去中心化的构建方式降低了技术门槛，确保用户无需手动修改复杂的文件夹结构即可完成团队扩容。

### 第一阶段：初识标杆 AI 团队成员

在动手创建前，参考下表设定可帮助你规划团队职能：

| Agent 名称        | 角色定义   | 工作模式    | 核心任务与特性                                       |
| ----------------- | ---------- | ----------- | ---------------------------------------------------- |
| **Gock (公鸡)**   | 情报员     | 定时/交互   | 搜索 AI 进展；支持对特定话题的“深挖”递归搜索。       |
| **Otter (河狸)**  | 私人助理   | 24h/定时    | 替代笔记软件；早间天气提醒；语音备忘与待办管理。     |
| **May (狮子)**    | 视频流水线 | 定时/指令   | 监控 X/YouTube 热点；通过 **[Thumbs Up]** 推进脚本。 |
| **Pandas (熊猫)** | 开发者     | 24/7 不间断 | 自动化编码、测试边缘情况、寻找系统 Bug。             |
| **Monkey (猴子)** | 内容转译员 | 被动执行    | 视频链接转文章；自动化网页截图与脚本重写。           |
| **Tiger (老虎)**  | 安全审计员 | 定时任务    | 扫描隐患；评估版本更新对当前任务的性能影响。         |

### 第二阶段：新建智能体 (Agent) 的标准化流程

推荐通过与“主智能体”对话的方式进行创建，以确保目录结构与参数配置的自动化生成：

1. 在 Telegram 中找到并打开 **[BotFather]**。
2. 点击或输入 **[/newbot]**，并为新智能体命名（如 Fox）。
3. 获取系统生成的 **TOKEN** 并点击 **[复制]**。
4. 回到 Telegram，与你的 **[主智能体]**（如 May）开启对话。
5. 发送指令：“新建一个 agent，它叫 [名称]，要做 [任务描述]”。
6. 将 **TOKEN** 粘贴并发送给主智能体，等待系统自动完成初始化。

### 第三阶段：深度解析 OpenClaw 的三层架构

理解底层逻辑是进行高级定制的前提，OpenClaw 的架构由全局、智能体、线程三层组成：

#### 1. 全局层 (Global Layer)

核心配置文件为 **OpenClaw.json**，它是整个系统的公共设施层。包含以下关键参数：

- **auth：** 定义模型厂商及登录身份。
- **agents：** 记录 Agent 列表、默认模型及运行参数。
- **tools：** 管理全局工具权限。
- **bindings：** 定义消息来源与 Agent 的映射关系。
- **browser：** 配置浏览器相关信息。
- **channel：** 管理 Telegram、Discord 等外部渠道连接。
- **Gateway：** 网关配置。
- **memory：** 全局记忆参数。
- **特别建议：** AI 在修改此文件时极易出错导致崩溃。务必在 Prompt 中强制要求：**[修改前必须提前通知并确认]**。

#### 2. 智能体层 (Agent Layer)

每个 Agent 在 Workspace 内拥有独立文件夹。加载上下文时，**[skills]** 文件夹内的工具优先级高于全局工具。核心定义文件包括：

- **agent.md：** 工作说明书（规则、优先级、边界）。
- **soul.md：** 性格与语气风格定义。
- **user.md：** 用户偏好与称呼习惯。
- **identity.md：** 身份卡（“我是谁”）。
- **tool.md：** 本地工具使用说明。
- **memory.md：** 提炼后的长期精华记忆（非原始对话记录）。
- **heartbeat.md：** 心跳文件，用于维持 Agent 状态。

#### 3. 对话线程层 (Session Layer)

决定 Agent 在特定窗口的工作状态：

- **[Slash new]：** 开启新线程，保留旧对话，适用于切换任务。
- **[Slash reset]：** 彻底清空上下文，是解决逻辑混乱或调试死循环的必要手段。

## 4. 【高阶技巧与避坑指南】：效能最大化策略

多 Agent 协作需平衡“并行效率”与“记忆纯净度”。

### 协作模式分析

- **固定分工（长期团队）：** 适用于业务边界清晰的任务流，如研究、写作、审核的串联。
- **临时子代理 (Sub-agent/Sessions spawn)：** 适用于“重研究任务”。
  - **案例：** 用户可以指令主 Agent 通过 **[sessions spawn]** 在后台研究“OpenClaw 讨论热点”，而主窗口继续处理“今日天气”查询。子代理完成后会主动汇报，实现无阻塞并行。

### 性能优化（顾问建议）

- **文件精简原则：** 务必保持 **agent.md** 和 **heartbeat.md** 等文件短小。文件长度直接影响 Token 成本与推理速度，尤其在多 Agent 运行环境下，冗长的配置文件会导致成本指数级增长。
- **渐进式构建：** 严禁一次性搭建庞杂团队。应从单一主 Agent 开始，跑通逻辑后再按需增加专业角色。

### 避坑准则

- **识别“非必要”拆分：** 对于短问题或上下文高度连续的任务，强行拆分 Agent 会让系统变重变慢。
- **防止记忆污染：** 长期记忆应在 **memory.md** 中隔离。避免让单一 Agent 处理过多异质任务，防止不同领域的逻辑在长期记忆中混淆。

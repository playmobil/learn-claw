

# Imagez × 微信回图打通记录

> 目标：在微信里输入 `#imagez 兔子`，自动生成图片，并把兔子图片直接回到当前微信对话框。

---

## 一、需求背景

最开始已经有一个基础的文生图 Python 示例，核心能力是：

- 调图片生成 API
- 处理 `b64_json` / `url` 两种返回格式
- 把图片保存到本地

但这还只是“生成图片文件”。

真正的目标是更进一步：

1. 在 OpenClaw 里做成一个可复用的 `imagez` skill
2. 能在当前对话里通过命令触发
3. 最终把图片作为**微信图片消息**发回去，而不是只给一个本地路径

也就是说，这不是单纯做个脚本，而是要把：

- 图片生成
- skill 封装
- 上层输出协议
- 微信消息回传
- OpenClaw 路由接入

这一整条链打通。

## 四、第三阶段：为 OpenClaw / 微信回复链路设计媒体协议

到这一步，图片已经能生成，也有了适合程序消费的结构。

但要想让微信里真正“看到图”，还差一个关键层：

> 上层必须知道这次返回的不只是文本，而是一条带图片的回复。

### 1. 新增包装脚本

新增了：

- `skills/imagez/scripts/emit_openclaw_media_reply.py`

这个脚本做的事是：

1. 调用 `generate_image.py`
2. 解析结果
3. 输出 OpenClaw 更容易消费的结构

例如：

```json
{
  "ok": true,
  "reply": {
    "text": "兔子来啦。",
    "mediaUrl": "/absolute/path/output/image_xxx.jpg",
    "mediaUrls": ["/absolute/path/output/image_xxx.jpg"]
  },
  "artifacts": {
    "primary_image": "...",
    "attachments": [...],
    "files": [...],
    "output_dir": "..."
  },
  "generator": {
    "model": "grok-imagine-1.0",
    "size": "1024x1024",
    "count": 1,
    "prompt": "一只可爱的兔子"
  }
}
```

### 2. 微信集成的关键结论

查了 OpenClaw 本地文档和微信插件实现后，确认了一件非常关键的事：

> 微信插件发媒体消息时，不是只靠 `userId` 就行，它还依赖**当前入站消息的 contextToken**。

这意味着：

- 脱离当前会话上下文的裸 CLI 发图不可靠
- 想回图，必须走“当前这条微信消息的回复链路”

### 3. 写集成文档

- `skills/imagez/references/openclaw-weixin-integration.md`

## 六、第五阶段：给微信插件接入 `#imagez`

为了让用户直接在微信里输入：

```text
#imagez 兔子
```

就得到图片回复，开始接入微信插件。

### 1. 确定最佳接入点

没有去改主 agent prompt，而是直接查微信插件代码，定位到：

- `src/messaging/process-message.ts`
- `src/messaging/slash-commands.ts`

发现现有插件已经有“短路命令处理”能力，最省事的办法是：

- 在消息处理阶段识别 `#imagez ...`
- 直接调用 imagez 包装脚本
- 然后复用插件现成的媒体发送能力把图回到当前会话

### 2. 新增 `#imagez` 分支

在 `slash-commands.ts` 中新增了：

- 解析 `#imagez <prompt>`
- 构造更适合出图的提示词
- 调用 `emit_openclaw_media_reply.py`
- 读取 `reply.mediaUrl`
- 再调用 `sendWeixinMediaFile(...)`

同时保留文本提示，比如：

- “正在生成图片，稍等一下。”
- “给你生成好了。”

### 3. 入口 bug：为什么一开始完全没命中

第一次改完后发现：

- `#imagez 兔子` 发出去没有触发新逻辑
- 日志里看不到 `imagez command`

排查后发现不是插件没加载，而是：

> `process-message.ts` 只允许以 `/` 开头的内容进入 `handleSlashCommand`

所以虽然 `slash-commands.ts` 写了 `#imagez`，但入口压根没放行。

### 4. 修入口判定

把入口条件从：

```ts
textBody.startsWith("/")
```

改成同时支持：

```ts
textBody.startsWith("/") || textBody.trim().toLowerCase().startsWith("#imagez")
```

这一步修完后，`#imagez` 终于能命中命令分支。

## 七、第六阶段：最难的一环——微信 CDN 上传失败

当 `#imagez` 终于被正确命中后，链路继续往下执行，说明：

- 命令识别成功
- 图片生成成功
- 本地图片文件存在
- 微信媒体发送函数也被调用到了

但还是没有回图。

### 1. 日志排查

继续查网关日志，关键报错是：

- `sendWeixinMediaFile` 已经拿到本地 jpg
- 进入 `uploadFileToWeixin`
- 之后连续 3 次：
  - `TypeError: fetch failed`

也就是说，失败点不在 imagez，不在 prompt，不在路由，而是在：

> 微信图片上传到 CDN 这一步

### 2. 为什么这一步特别难排查

因为前面一切看起来都像“差不多通了”：

- 命令触发了
- 图生成了
- 本地文件也有了

但只要 CDN 上传不成功，微信就收不到图片。

所以这时候必须往更底层去看：

- `send-media.ts`
- `cdn/upload.ts`
- `cdn/cdn-upload.ts`
- `api/api.ts`

把微信插件内部上传流程翻出来看。

### 3. 发现的关键问题：CDN 地址被写死错了

排查后发现一个很致命的点：

在给 `#imagez` 接媒体回传时，代码里一度把 CDN 地址写死成了：

```ts
https://openclaw-cdn.clawd.cloud
```

但微信账户解析里，正常应该使用的是账号对应的 CDN 配置，默认值是：

```ts
https://novac2c.cdn.weixin.qq.com/c2c
```

这意味着前面实际上出现了一个很隐蔽但很致命的问题：

- 上传票据是从微信 API 拿的
- 但真正上传时，却发到了另一个硬编码 CDN 地址

这种情况最容易出现的结果就是：

- `fetch failed`
- 或握手失败
- 或上传协议不匹配

### 4. 修复方法

于是把 `slash-commands.ts` 里的写死 CDN 去掉，改成：

- 在 `SlashCommandContext` 里新增 `cdnBaseUrl`
- 在 `process-message.ts` 里把账号解析得到的 `deps.cdnBaseUrl` 传进去
- 最终让 `sendWeixinMediaFile(...)` 使用：

```ts
ctx.cdnBaseUrl || "https://novac2c.cdn.weixin.qq.com/c2c"
```

这样 `#imagez` 就不再绕开账号配置，而是走微信插件原本的 CDN 通道。

### 5. 重启网关验证

修复完之后重新执行：

- `openclaw gateway restart`

再回到微信重试。

最终验证结果：

> 现在可以了。

也就是说，最后真正卡住整条链的，不是 imagez、不是 prompt、不是 mediaUrl 协议，而是 **微信 CDN 上传时走错了 base URL**。

## 八、最终实现结果

现在整条链已经打通，效果是：

### 用户在微信发送

```text
#imagez 兔子
```

### 系统执行流程

1. 微信插件收到消息
2. `process-message.ts` 识别到 `#imagez`
3. 进入 `handleSlashCommand(...)`
4. 命中 `#imagez` 分支
5. 调用 `emit_openclaw_media_reply.py`
6. 包装脚本再调用 `generate_image.py`
7. imagez 生成本地图片
8. 返回 `reply.mediaUrl`
9. 微信插件调用 `sendWeixinMediaFile(...)`
10. 图片上传到正确的微信 CDN
11. 以图片消息形式发回当前微信对话

最终用户在对话框里直接看到兔子图片，而不是只有一串文件路径。

## 十、关键经验总结

这次真正麻烦的，不是“生成图片”，而是“把生成的图片变成微信里的一条原生图片消息”。

里面最关键的经验有几条：

### 1. 文生图成功 ≠ 聊天发图成功

图片能生成，只代表本地文件落地成功。

要在微信里看到图，还需要：

- 上层识别这是媒体回复
- 当前会话上下文保留 `contextToken`
- 微信插件能上传到正确 CDN
- 再走正确的发送接口

任何一环错了，最终都只是“本地有张 jpg，但用户看不到”。

### 2. `contextToken` 是微信回图链路里的关键

脱离当前入站会话上下文，单靠 `userId` 并不足以稳定回媒体消息。

所以这种能力最适合做成：

- 当前消息触发
- 当前消息上下文内直接回复

而不是做成一个完全脱离上下文的裸发送脚本。

### 3. 插件入口判定很容易成为隐蔽阻塞点

如果消息入口只放行 `/...`，那你在下游写再多 `#imagez` 逻辑也不会被触发。

这类问题看起来像“功能没生效”，本质却是“根本没进分支”。

### 4. 最后那类 `fetch failed` 往往不是业务逻辑，而是底层网络/地址/协议问题

这次就是典型例子：

- 业务层已经走通
- 生成层已经走通
- 文件层已经走通
- 但底层 CDN 地址不对

于是最后会表现成一句非常不友好的：

- `TypeError: fetch failed`

不盯底层实现，很难直接猜到根因。
